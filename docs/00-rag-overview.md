# 个人知识库 RAG 总体设计

## 1. 项目目标

本项目面向个人知识库场景，核心数据源以 PDF 为主，目标是构建一个既能真实使用、又能作为面试展示项目的完整 RAG 系统。

项目重点不只是实现基础问答，而是完整体现以下能力：

- 非结构化 PDF 的结构化解析
- 多路检索与重排优化
- 基于证据的答案生成与引用返回
- 面向工程落地的评测、观测与调优闭环

当前约束与选型如下：

- PDF 解析：`MinerU API`
- 图片理解：`多模态 LLM API`
- 最终生成模型：`DeepSeek API`
- 向量数据库：`ChromaDB`
- 系统定位：`展示优先，可接 API，链路完整`

## 2. 总体架构

考虑到当前使用方式是“指定 PDF 文件夹路径，直接构建向量数据库”，因此不单独设计上传式数据接入层。整体架构收敛为 3 个核心层：

1. 离线索引构建层
2. 在线检索与生成层
3. 评测与观测层

整体主流程如下：

```text
指定PDF目录
  -> MinerU解析PDF
  -> 图片/图表理解
  -> 文本清洗与标准化
  -> 结构感知Chunk切分
  -> Embedding生成
  -> 关键词索引构建
  -> Chroma向量库写入
  -> 在线Query处理
  -> 多路召回
  -> RRF融合
  -> Cross-Encoder精排
  -> 上下文构造
  -> DeepSeek生成答案
  -> 返回答案与引用
  -> 记录日志并进入评测/观测闭环
```

## 3. 三层设计摘要

### 3.1 离线索引构建层

负责将 PDF 转化为高质量、可检索、可追溯的知识表示。

主要职责：

- 扫描指定 PDF 文件夹
- 调用 MinerU 解析 PDF
- 调用多模态 LLM 理解图片、图表、表格截图
- 对解析结果做清洗与标准化
- 基于文档结构进行 chunk 切分
- 生成 embedding
- 构建关键词索引
- 将 chunk 与 metadata 写入 ChromaDB

### 3.2 在线检索与生成层

负责将用户问题映射为高质量证据，并基于证据生成可解释答案。

主要职责：

- Query 理解与轻量改写
- 检索路由与多路召回
- 语义检索与关键词检索融合
- RRF 融合排序
- Cross-Encoder 精排
- 上下文拼装
- DeepSeek 生成答案
- 输出引用、证据块与调试信息

### 3.3 评测与观测层

负责持续验证系统效果，而不是只停留在“能跑通”。

主要职责：

- 构建离线评测集
- 计算召回质量指标
- 计算答案质量指标
- 记录延迟与 token 成本
- 保存 query 处理链路
- 支持 case 回放与错误分析

## 4. 关键设计原则

### 4.1 结构优先，不做纯文本式 PDF RAG

PDF RAG 的核心难点不在 LLM，而在文档结构恢复。系统应尽量保留：

- 标题层级
- 段落边界
- 表格内容
- 图片说明
- 页码与章节路径

### 4.2 原始 Query 不应被改写覆盖

在线检索阶段不应只使用改写后的 query，而应保留：

- 原始 query
- 检索友好改写 query
- 关键词增强或子问题 query

这些 query 并行进入不同召回通道，再做融合。

### 4.3 多路召回先于精排

正确顺序应为：

`query处理 -> 多路召回 -> RRF融合 -> 候选集缩减 -> Cross-Encoder精排 -> 上下文构造 -> 生成答案`

### 4.4 Cross-Encoder 只用于小候选集

Cross-Encoder 适合对 `top 10 ~ top 20` 候选做高精度重排，不适合直接对全库检索。

### 4.5 评测必须内建

面试展示型 RAG 的关键不是“我做了很多模块”，而是“我能说明这些模块为什么存在、带来了什么收益、如何衡量收益”。

## 5. 推荐迭代路线

### Phase 1：最小可用闭环

- 指定 PDF 文件夹
- MinerU 解析
- 文本清洗
- chunk 切分
- embedding 生成
- Chroma 检索
- DeepSeek 回答

目标：完成端到端闭环。

### Phase 2：完整检索链路

- 加入 BM25 或 FTS 关键词检索
- 引入多 query 召回
- 加入 RRF 融合
- 引入 Cross-Encoder 精排
- 返回引用与证据

目标：形成完整工程型 RAG。

### Phase 3：展示与优化

- 图片理解增强
- 离线评测集
- 检索与生成指标体系
- debug 页面
- 延迟与成本统计
- case 回放

目标：形成可展示、可讲解、可持续优化的版本。

## 7. 面试表达主线

这个项目最适合的讲述方式不是“我搭了一个问答机器人”，而是：

1. 我把以 PDF 为主的个人资料做了结构化知识化处理
2. 我设计了多路召回与重排链路，提升检索质量
3. 我让系统支持引用返回、调试链路和指标评测
4. 我把它做成一个可持续优化的 RAG 工程系统

这条主线比单纯展示“用了哪些 API”更有说服力。
