# 在线检索与生成层设计

## 1. 目标

在线检索与生成层负责将用户问题转化为高质量证据，并基于证据生成可靠答案。

这一层的核心目标不是“让模型直接回答”，而是：

1. 正确理解问题类型
2. 从知识库中找到尽可能相关的证据
3. 通过融合与重排提升候选质量
4. 为 LLM 构造清晰、紧凑、可引用的上下文
5. 输出带依据的答案

## 2. 在线主流程

推荐主流程如下：

```text
用户Query
  -> Query理解
  -> Query轻量改写 / 扩展 / 分解
  -> 检索路由
  -> 多路召回
  -> RRF融合
  -> 候选去重与粗排
  -> Cross-Encoder精排
  -> 上下文构造
  -> DeepSeek生成答案
  -> 输出答案、引用、调试信息
```

## 3. Query 理解

### 3.1 为什么需要

不同问题适合不同检索与生成策略。

例如：

- 事实查找型问题更依赖精确召回
- 总结归纳型问题更依赖上下文完整性
- 跨文档对比更依赖多证据覆盖
- 写作辅助更依赖上下文组织和生成约束

### 3.2 推荐意图分类

建议至少识别以下类型：

- `fact_lookup`
- `single_doc_summary`
- `multi_doc_comparison`
- `reasoning_synthesis`
- `writing_assistant`

### 3.3 Query 理解输出

Query analyzer 可以输出：

- `intent_type`
- `need_quote`
- `need_multi_doc`
- `need_image_evidence`
- `need_table_evidence`
- `retrieval_mode`

## 4. Query 改写策略

### 4.1 query 改写的原因

用户的query存在的问题，主要有两个方面：
- 信息缺失，用户表达的不全面；
- 噪声问题，用户query里面携带了很多不相关的东西。

### 4.2 信息缺失的情况

- 1. 历史会话改写
在会话中，上下文是有关联的，如果仅仅使用query进行召回，会导致召回精确度大幅下降，因为query中缺失了很多会话信息，如下所示。

```text
• 用户：iPhone 15 的性能怎么样？
• 系统：iPhone 15 的性能非常强大，搭载了 A17 芯片，支持更高效的图像处理和长时间续航。
• 用户：它和上一代相比有哪些提升？
• 系统：iPhone 15 相较 iPhone 14 提升主要体现在芯片性能、相机优化和电池续航上。
• 用户：相机的具体改进是什么？

改写前：相机的具体改进是什么？
改写后：iPhone 15 的相机相比 iPhone 14 有哪些具体改进？
```

- 2. 关键词扩写
用户在使用搜索引擎的时候，经常会输入一些关键词，关键词比较少，而且缺乏语义，在进行语义检索（向量检索）的时候，召回不是特别好，所以需要对用户的关键词进行补充完善。

```text
用户输入：
“Python 数据分析”
改写后的 Query：
“Python 在数据分析中的应用是什么？有哪些常用的库和工具？”
```

- 3. 伪答案改写
它通过在原始查询中加入一个伪答案的形式，来丰富查询的语义，提升检索或回答的准确性。伪答案并非实际答案，而是一个假设性的内容，用于引导系统检索更相关的信息。

```text
用户输入：
“初创公司如何融资？”
改写后的 Query：
“初创公司如何融资？比如通过风险投资、众筹或银行贷款等方式实现资金筹集。”
```

伪答案目的：提供假设性的融资方式，增加语义信息，提升对复杂问题的检索能力。

- 4. 缩写词改写
用户进行查询的时候，有时会使用缩写，但是很多相关的文档里面一般是全写，所以需要进行改写，完善缩写的全称。

```text
用户输入：
“AI 在医疗中的应用”
改写后的 Query：
“人工智能（Artificial Intelligence）在医疗中的应用是什么？有哪些具体案例？”
```
### 4.3 存在噪声问题的情况

- 1. 一般去噪改写
它会去除查询中的噪声（例如无关的词汇、模糊的表达、多余的修饰等），以简化和优化查询，使其更具可操作性。这种方法对提升检索精度和效率非常有帮助。

```text
用户输入：
“孩子最近在学校数学成绩不太好，怎么才能找到一些比较实用的方法，帮助他提高成绩呢？”
改写后的 Query：
“有哪些实用方法可以帮助学生提高数学成绩？”
```

分析：去掉与问题无关的背景信息 “孩子最近在学校数学成绩不太好”，直接提取核心意图 “帮助提高数学成绩”。

- 2. 关键词改写
它是一种专注于提取核心关键词并去除噪声的 Query Rewriting 方法。这种方法通过识别查询中的关键部分，排除冗余信息（如停用词、语气词、多余描述），使查询更简洁，从而提升检索效率和准确性。这种提取关键词的方式，比较适合用于关键词检索召回中，比如bm25检索召回。

```text
用户输入：
“关于 Python 的并发处理，有哪些常用的库？”
改写后：
“Python 并发处理 常用库”
```

- 3. 子查询改写
当查询是对比类的查询时，确实可能会出现多个实体相互干扰的情况。对比类查询通常包含多个元素，直接查询时，这些元素可能会造成噪声或信息重叠，影响检索的精确性。为了减少干扰信息，可以将对比类查询拆分成多个单独的查询，每个查询聚焦于一个实体，从而避免信息混淆，得到更精准的结果。

```text
用户输入：
“Python 和 Java 哪个更适合开发人工智能？”
拆分后的查询：
“Python 适合开发人工智能的优点有哪些？”
“Java 适合开发人工智能的优点有哪些？”
```

拆分原因：
直接询问 Python 和 Java 的对比可能会让系统无法准确判断其各自的特点。拆分后，系统可以分别检索 Python 和 Java 在人工智能开发中的应用，避免相互干扰。

### 4.4 关键原则

不要让改写后的 query 覆盖原 query。推荐保留多路 query 并行召回。

因为改写可能带来语义漂移，尤其在：

- 用户问题本身很具体时
- 问题包含专有名词时
- 问题带有个人表达风格时

因此原 query 必须保留。

### 4.5 改写方式

第一版建议只做轻量改写：

- 补足省略主语
- 明确核心概念
- 扩展同义词或术语
- 关键词扩写
- 一般去噪改写
- 子查询改写

复杂 query 才考虑：

- query decomposition
- multi-hop 子问题生成

暂时不考虑：

- 历史会话改写

### 4.6 prompt 设计

```text
你是查询扩展方面的专家，能够生成问题的释义。

我无法直接使用用户的问题从知识库中检索相关信息。

你需要通过多种方式扩展或释义用户的问题，例如使用同义词/短语、完整地写出缩写、添加一些额外的描述或解释、改变表达方式、将原始问题翻译成另一种语言（英语/中文）等。

并返回 3 ~ 5 个版本的问题，其中一个来自翻译。

只需列出问题。不需要其他单词。
```

## 5. 检索路由

检索路由的作用是根据 query 类型选择合适召回路径，而不是所有问题都用完全相同的流程。

### 5.1 路由示例

- 事实查找：偏 `small chunk + sparse + dense`
- 总结归纳：偏 `large chunk + dense`
- 多文档对比：偏 `dense + sparse + broader topk`
- 表格问题：优先召回表格摘要 chunk
- 图片问题：优先召回图片理解 chunk

### 5.2 第一版建议

第一版不必做太复杂的路由策略，可以先做：

- 默认 dense + sparse 双路召回
- 根据意图调整 chunk 粒度和 topk

## 6. 多路召回设计

这是在线层的核心模块。

### 6.1 Dense Retrieval

基于 embedding 进行语义检索。

适合：

- 概念表达不完全一致的问题
- 语义相近但措辞不同的问题
- 总结类、理解类问题

### 6.2 Sparse Retrieval

基于 BM25 或 FTS 的关键词检索。

适合：

- 专有名词
- 术语
- 缩写
- 公式名称
- 精确匹配类问题

### 6.3 Metadata Filter

当 query 中包含显式限制条件时，可加 metadata 过滤：

- 指定文档名
- 指定标签
- 指定主题
- 指定时间范围

## 7. 融合策略

## 7.1 为什么用 RRF

不同召回器分数空间不可直接比较，因此第一版推荐使用 `RRF` 融合。

RRF 的优点：

- 实现简单
- 稳定性高
- 对 dense / sparse 混合场景友好

### 7.2 融合流程

建议各通道先取：

- dense `top 20 ~ 30`
- sparse `top 20 ~ 30`
- 特殊路由召回 `top 10 ~ 20`

然后统一用 RRF 融合，保留候选集 `top 15 ~ 20`。

以上 top 参数需要设计成可单独设置

### 7.3 融合后的处理

RRF 后需继续做：

- 去重
- 同文档相邻 chunk 合并
- 类型平衡
- 明显低相关样本剔除

## 8. 重排设计

重排建议分两层。

### 8.1 粗排

在融合后先做简单规则调整，例如：

- 标题命中加权
- 文档标题命中加权
- 页码邻近加权
- 图片/表格 chunk 按意图加权
- 同章节多个命中时适度提升

### 8.2 精排

使用 Cross-Encoder 对融合后的候选集做高精度重排。

推荐流程：

- 候选集 `top 15 ~ 20`
- Cross-Encoder 重排
- 输出 `top 3 ~ 5`

以上 top 参数需要设计成可单独设置

### 8.3 为什么 Cross-Encoder 放最后

因为 Cross-Encoder 精度高但成本也高，只适合在小规模候选上使用。

## 9. 上下文构造

上下文构造决定了生成质量，不应把 topk 结果直接拼接给 LLM。

### 9.1 上下文组成

每条证据建议包含：

- 文档标题
- 章节路径
- 页码
- chunk 正文
- chunk 摘要
- 图片/表格附加说明

### 9.2 组织策略

建议优先按“主题完整性”而不是“纯分数顺序”组织：

- 同章节内容可适度合并
- 强相关 chunk 放前面
- 重复信息去重
- 控制总 token 长度

### 9.3 不同问题的上下文策略

- 事实查找：少量高精度证据
- 总结归纳：较完整的同主题证据
- 跨文档对比：每篇文档至少保留关键证据
- 写作辅助：保留观点与出处的组合

## 10. 答案生成设计

### 10.1 生成模型

最终生成使用 `qwen-plus API`。

### 10.2 Prompt 约束

生成 prompt 需要明确约束：

- 只基于提供证据回答
- 证据不足时明确说明
- 优先给出引用
- 区分“原文信息”和“综合推断”
- 禁止无依据扩写

### 10.3 输出结构

建议返回结构化结果：

- `answer`
- `citations`
- `used_chunks`
- `confidence`
- `retrieval_debug_info`

### 10.4 引用设计

引用建议最少精确到：

- 文档标题
- 页码
- 章节路径

这样更利于个人知识库回溯，也更适合展示。

## 11. 对话增强能力

如果后续支持多轮问答，建议增加：

- 对话历史摘要
- 当前轮 query 改写时纳入历史上下文
- 历史主题对齐

但第一版不要过早引入复杂对话记忆，否则会增加调试难度。

## 12. 常见失败模式

### 12.1 检索失败

- 只靠 dense，导致术语问题表现差
- 只靠 sparse，导致语义类问题召回不足
- 改写过度，偏离原始问题
- chunk 太碎，证据不完整

### 12.2 重排失败

- 候选集太小，正确答案在召回阶段已经丢失
- 候选集太大，精排成本过高
- 没有区分 chunk 类型，表格/图片证据被压制

### 12.3 生成失败

- 上下文碎片化
- 引用和答案不一致
- 证据不足但模型仍然强行回答

## 13. 推荐第一版实现

第一版在线层建议做以下最小闭环：

1. Query analyzer 做轻量意图识别
2. 保留原 query，并生成 1 个检索友好改写 query
3. dense + sparse 双路召回
4. 用 RRF 融合
5. 对 top 15 做 Cross-Encoder 精排
6. 选 top 3 ~ top 5 做上下文构造
7. 用 DeepSeek 生成带引用答案

## 14. 后续增强方向

- 多 query 扩展
- query decomposition
- 基于问题类型的动态 topk
- 针对表格和图片的专用重排
- answer grounding 检查
- 引用一致性自动验证
- 输出答案摘要与证据摘要分离
